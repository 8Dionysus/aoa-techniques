# multi-source-primary-input-provenance Checklist

- [ ] The bridge or combined docs surface names one primary input.
- [ ] The supporting inputs are named explicitly and remain secondary to the primary input.
- [ ] The surface makes clear what the `primary` label authorizes and what it does not authorize.
- [ ] Supporting inputs are contextual or corroborating inputs, not silent co-primary anchors.
- [ ] The priority order is visible in authored markdown.
- [ ] Downstream readers can preserve the order without needing graph semantics.
- [ ] The bundle does not rely on note provenance handles to explain source priority.
- [ ] The bundle does not widen into retrieval ranking or traversal policy.
- [ ] The bundle does not become broad bridge architecture doctrine.
- [ ] The example still makes the primary-versus-supporting split obvious at a glance.
- [ ] If a supporting input conflicts with the primary input, the surface flags review instead of silently overriding or re-ranking authority.
