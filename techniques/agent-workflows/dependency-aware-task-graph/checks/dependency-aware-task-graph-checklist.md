# dependency-aware-task-graph checklist

Use this checklist when reviewing whether a dependency-aware task graph is still a bounded working surface instead of a drifting tracker or memory layer.

- [ ] each node maps to one reviewable work slice rather than a vague project theme
- [ ] dependency edges describe real blockers instead of narrative ordering preferences
- [ ] blocked tasks name the unmet prerequisite that keeps them closed
- [ ] ready work is derived from the current graph state rather than guessed from chat history
- [ ] node state is updated when real work completes or blocker state changes
- [ ] the graph stays smaller than a full project-management system or memory substrate
- [ ] implementation review, validation, and release checks still live outside the graph
- [ ] at least one example shows the ready frontier changing when a blocker is cleared
