# Technique Selection Guide

This guide defines the bounded contract behind the selection family.

Use it when the question is not which one technique to choose, but how the repo's current selection surfaces are allowed to route that choice without turning into scoring, ranking, or graph behavior.

See also:
- [Start Here](START_HERE.md)
- [Technique Selection](TECHNIQUE_SELECTION.md)
- [Selection Patterns](SELECTION_PATTERNS.md)
- [`../generated/technique_catalog.json`](../generated/technique_catalog.json)
- [Documentation Map](README.md)

This family uses one stable shape:

- authoritative source:
  - `TECHNIQUE.md` frontmatter
  - the current generated catalog projection
  - validator-backed domain-start specs, common-move specs, and review-backed working-set specs
- reader companions:
  - `TECHNIQUE_SELECTION.md`
  - `SELECTION_PATTERNS.md`
- adjacent but separate family:
  - `SHADOW_PATTERNS.md` stays under `TECHNIQUE_SHADOW_GUIDE.md`, even though it is generated by the same build step
- derived manifests:
  - `generated/technique_catalog.json`
  - `generated/technique_catalog.min.json`
- what it must not become:
  - scoring
  - ranking
  - policy routing
  - graph traversal

## Source Contract

The selection family stays bounded to shallow bundle-routing knowledge:

- `id`
- `name`
- `domain`
- `status`
- `summary`
- current review-related metadata fields
- direct typed `relations`
- validator-backed starting points, common moves, and review-backed working sets

The family is allowed to help a reader choose one next technique or one nearby cluster.
It is not allowed to replace `TECHNIQUE.md`, semantic review docs, or bundle-local notes as the source of meaning.

If a domain has no canonical default yet, the validator-backed domain-start specs may point to one bounded `promoted` starter for that domain.
That is a temporary route aid, not a silent promotion.

## Reader Companions And Derived Catalog

The current selection family exposes four bounded surfaces:

- `TECHNIQUE_SELECTION.md`
  - use when one bounded first-pick choice is enough
- `SELECTION_PATTERNS.md`
  - use when the flat catalog is not enough and the next step needs validator-backed starts, common moves, or review-backed working sets
- `generated/technique_catalog.json`
  - the full generated catalog payload
- `generated/technique_catalog.min.json`
  - the strict min projection for smaller routing consumers

`SHADOW_PATTERNS.md` remains adjacent, but separate. It is a canonical-only caution lookup surface and should not be read as part of the selection-family contract.

## Boundaries

- Selection stays bundle-level and one-step only.
- Direct `relations` are adjacency hints, not graph inference.
- Working sets stay review-backed and bounded to authored review docs.
- The generated catalog and readers do not become a policy engine, a scoring layer, or a replacement for reading the bundle itself.

## Regeneration

When selection-family sources change, regenerate and validate with:

- `python -m pip install -r requirements-dev.txt`
- `python scripts/build_catalog.py`
- `python scripts/validate_repo.py`
- `python scripts/release_check.py`
